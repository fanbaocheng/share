推理性能：QPM 速算公式 Decode 阶段 · 带宽瓶颈

给定模型参数与 GPU 型号，速算单节点能支撑的 QPM（每分钟查询数）。

3.1   核心推导逻辑

LLM 推理分两个阶段：

Prefill（预填充）：一次性处理输入 prompt，计算密集型，瓶颈在 FLOPs。

Decode（解码）：逐 token 自回归生成，每步都要重新读取全部权重+KV Cache，内存带宽密集型。

实际在线服务中，Decode 阶段占生成总耗时的 80-95%，因此 QPM 的瓶颈几乎总是内存带宽。

3.2   显存容量约束（能装下吗？）

单节点总显存  =  N_gpu × Mem_gpu

推理所需显存  =  M_weights + M_kv + M_overhead

M_weights  =  P_total × B_param   // MoE 模型：全部参数都要加载到显存
M_kv  =  2 × L × H_kv × D_head × S × B_concurrent × B_kv   // 2=K+V
M_overhead  ≈  2~4 GB   // CUDA 上下文 + 框架缓冲

推理（decode 阶段）的中间激活值是临时分配、前向完即释放的，框架内存池会复用，因此不占用稳态显存。激活值仅在 prefill 阶段处理长 prompt 时瞬态较大，但算完即释放，不影响"能否装下"的判断。

参数说明：

P_total模型总参数量（含 MoE 全部专家）如 1.6T（DeepSeek-V4-Pro）
B_param每参数字节数FP16=2, INT8=1, INT4=0.5
LTransformer 层数如 80
H_kvKV 头数（GQA 时较少）如 Llama3-70B 为 8
D_head每头维度通常 128
S上下文长度（token 数）可调变量，默认 4096

3.3   QPM 公式（Decode 阶段，min(带宽瓶颈, 计算瓶颈)）

▎ 终极公式：单节点 QPM（Decode 阶段，min(带宽瓶颈, 计算瓶颈)）

// 核心：Decode 每步读取 = 权重 + KV Cache
tokens/sec  =  N_gpu × BW_mem × η_hw × η_comm × η_moe  /  (P_active × B_param + KV_per_token × S_avg)

QPM  =  tokens/sec × 60  /  S_out

// 其中：
P_active  =  每 token 活跃参数量   // Dense: P_total；MoE: 激活专家参数量
KV_per_token  =  2 × L × H_kv × D_head × B_kv   // 每 token 的 KV Cache 字节数
S_avg  =  S_ctx + S_out/2   // 平均 KV 序列长度（prompt + 半程输出）
S_ctx  =  prompt 上下文长度   // 改上下文会改变 KV 读取量
S_out  =  平均每请求输出 token 数
B_param  =  每参数字节数   // FP16=2, INT8=1, INT4=0.5
η_hw  =  0.45~0.65   // 算子利用率（vLLM+FlashAttn ≈ 0.55）
η_comm  =  1 - (N_gpu-1) / (N_gpu × R_nvlink × 2)   // 多卡通信效率（有 NVLink 时）
// R_nvlink = 显存带宽 / NVLink 带宽（H100: 3350/900≈3.72; A800: 2000/400=5）
// 无 NVLink（PCIe）: η_comm = 1 - 0.3×(N_gpu-1)/N_gpu（经验拟合）
η_moe  =  Dense=1.0, MoE=0.60~0.85   // All-to-All 通信 + 专家负载不均 + 路由开销
▎ 计算瓶颈公式（GPU 算不动时）

// 计算瓶颈：受限于 GPU FLOPS，而非带宽
tokens/sec  =  N_gpu × FLOPS × η_hw  /  (P_active × 1e9 × 2)

FLOPS  =  TFLOPS × 1e12   // GPU 标称 TFLOPS 转 FLOPS（×1e12）
每 token FLOPs  ≈  P_active × 1e9 × 2   // 前向 matmul ≈ 2×参数量

// 实际瓶颈 = min(带宽瓶颈, 计算瓶颈)
实际 tokens/sec  =  min(带宽瓶颈, 计算瓶颈)

经验系数说明

η_hw = 0.45~0.65   实测算子利用率。理论带宽不可能 100% 利用——FlashAttention 可达 60-70%，但 Continuous Batching、调度开销等因素拉低至 45-65%。vLLM 默认配置实测约 50-55%。取值建议：vLLM + FlashAttn 取 0.55，裸 PyTorch 取 0.40。

η_comm   多卡张量并行的通信效率。每步 AllReduce 需传 2×(N-1)/N 的激活数据，通信耗时与 NVLink/显存带宽比相关。8 卡 H100(NVLink 900GB/s vs 3.35TB/s, R=3.72) η_comm ≈ 0.88；8 卡 A800(NVLink 400GB/s vs 2TB/s, R=5.0) η_comm ≈ 0.91；8 卡 H20(NVLink 900GB/s vs 4TB/s, R=4.44) η_comm ≈ 0.90。R 越大表示内存带宽相对 NVLink 越快，通信开销相对越小。

η_moe = 0.60~0.85   MoE 模型综合效率损失。包含三个因素：① All-to-All 通信：token 路由到持有激活专家的 GPU，结果再回收，延迟与 EP 度正相关；② 专家负载不均：192 专家选 top-8，不是每 GPU 均匀分到，有的空转有的过载；③ 路由门控开销：每层都要计算 gate + dispatch，虽小但不可忽略。取值建议：EP=8 取 0.70，EP=4 取 0.80，纯 TP(无 EP) 取 0.90。

3.4   MoE vs Dense 的区别

Dense 模型MoE 模型
P_total= P_active（全部参与计算）≥ P_active（专家权重全部加载，但只激活部分）
M_weightsP_active × B_paramP_total × B_param（显存装全量）
每 token 计算 FLOPS2 × P_active2 × P_active（只算激活参数）
每 token 读取带宽P_active × B_paramP_active × B_param（只读激活专家权重）
+ 极小的路由门控开销（可忽略）

MoE 关键：显存瓶颈看 P_total（全部专家需加载），计算/带宽瓶颈看 P_active（每 token 只读激活专家）。Decode 阶段每 token 权重读取量 = P_active × B_param，与 Dense 模型形式相同，只是 P_active ≪ P_total。若启用 Expert Parallelism，不同专家分到不同 GPU，等效每 GPU 读取量进一步降低。

3.5   速算示例

场景 A：8×H20，Qwen3-32B（32B Dense），INT8 量化，4K 上下文，输出 500 token

每步权重读取 = 32B × 1 byte = 32 GB | KV读取 ≈ 0.5 GB（占 1.6%，可忽略）

η_comm = 1 - 7/(8×4.44×2) = 0.90   (R = 4000/900 = 4.44)

tokens/sec = 8 × 4000 × 0.55 × 0.90 / 32 ≈ 495

QPM = 495 × 60 / 500 ≈ 59 QPM

场景 B：8×H20，DeepSeek-V4-Flash（284B/13B MoE），INT8 量化，4K 上下文，输出 500 token

每步权重读取 = 13B × 1 byte = 13 GB | KV读取 ≈ 0.2 GB（MLA 压缩，可忽略）

η_comm = 0.90, η_moe = 0.70

tokens/sec = 8 × 4000 × 0.55 × 0.90 × 0.70 / 13 ≈ 853

QPM = 853 × 60 / 500 ≈ 102 QPM

MoE 的 P_active(13B) 远小于 Dense 的 P_total(32B)，即使乘以 η_moe=0.70 的损耗，每步读取量仍只有 Dense 的 28%，所以 MoE 的 QPM 反而更高。这是 MoE 推理的核心优势：以 4.4× 参数量换取 1.7× 吞吐。CSA/HCA 注意力压缩后 KV 几乎可忽略，长上下文场景优势更大。

训练性能：最低 GPU 数量公式 全量训练 · AdamW + Mixed Precision

给定模型参数与 GPU 型号，速算全量训练需要的最低显卡数量。

4.1   全量训练显存拆解

▎ 全量训练显存  =  M_model + M_grad + M_optim + M_act

M_model  =  P_total × 2 bytes   // FP16/BF16 权重
M_grad  =  P_total × 2 bytes   // 梯度（同精度）
M_optim  =  P_total × 8 bytes   // AdamW: m(4B) + v(4B) = 8B/param（BF16 推荐，不需 master）
// ⚠️ 如用 FP16 训练(需 master weights): 请 ×1.5 = 12B/param
M_act  =  S × B × H × L × 2 × 2 bytes   // 激活值(开启 activation checkpointing，只存输入)
// attention: 输入 S×B×H, FP16 = ×2 bytes | MLP: 输入 S×B×H, FP16 = ×2 bytes
// MoE 的 MLP 激活按 P_active/P_total 比例缩放

参数说明：

P_total模型总参数量MoE 按总参数计（含全部专家）
HHidden DimensionMoE 按激活参数的 hidden_dim
LTransformer 层数
S序列长度
B单 GPU Batch Size

4.2   最低 GPU 数公式

▎ 终极公式：最低 GPU 数

N_min  =  ⌈ M_train / (Mem_gpu × η_frag) ⌉

M_train  =  M_model + M_grad + M_optim + M_act   // ZeRO-0（无分片）
// ZeRO 分片后，M_grad 和 M_optim 按 N_gpu 均摊，详见 4.4

4.3   经验系数

η_frag = 0.85   显存碎片系数。实际可用显存 < GPU 标称值，CUDA 上下文 ~1.5GB、框架缓冲 ~1-2GB、碎片无法利用约 5-10%。综合取 0.85。

激活值(act ckpt)   公式中每层 ×2 表示：开启 activation checkpointing 后，每层只需存 attention 输入(2) + MLP 输入(2) = 2 项 × FP16(2 bytes) = 4 bytes/element。MoE 的 MLP 激活按 P_active/P_total 比例缩放。

4.4   ZeRO 策略显存节省

策略分片内容每 GPU 显存适用场景
ZeRO-0无分片M_total（完整副本）小模型 / 单卡
ZeRO-1优化器状态M_model + M_grad + M_optim/N中等模型
ZeRO-2优化器 + 梯度M_model + (M_grad+M_optim)/N大模型推荐
ZeRO-3全部分片(M_model+M_grad+M_optim)/N + M_act超大模型必需

ZeRO-3 的 M_act 不可分片（每 GPU 需完整激活），且额外通信开销约 15-20%。实际选择需平衡通信带宽与显存压力。

4.5   速算示例

场景：DeepSeek-V4-Flash（284B 总参/13B 激活），8×H20，ZeRO-3，4K 序列，BS=1

M_model = 284B × 2 = 568 GB | M_grad = 568 GB | M_optim = 284B × 8 = 2272 GB

M_act ≈ 4096 × 1 × 4096 × 43 × 2 × 2 ≈ 2.7 GB（MoE MLP 部分极小，可按全量估）

M_total = 568 + 568 + 2272 + 2.7 = 3410.7 GB

ZeRO-3: 每 GPU ≈ (M_model+M_grad+M_optim)/N + M_act = 3408/N + 2.7

解：3408/N + 2.7 ≤ 96×0.85 = 81.6 → N ≥ 3408/78.9 ≈ 43.2 → N_min = 44 卡（建议 48 卡 = 6 节点）

MoE 训练：权重/梯度/优化器按 P_total=284B 计算，激活按 hidden_dim=4096 计算。实际 ZeRO-3 还需通信缓冲区（prefetch+gradient reduce），实测约多 30-50% 开销，建议预留 64 卡（8 节点）。

LoRA 微调：最低 GPU 数量公式 低秩适配 · 冻结基座

给定模型参数与 GPU 型号，速算 LoRA 微调需要的最低显卡数量。

5.1   LoRA 显存拆解

▎ LoRA 微调显存  =  M_frozen + M_lora + M_lora_optim + M_lora_grad + M_act + M_overhead

M_frozen  =  P_total × B_base   // 冻结基座权重（可量化）
M_lora  =  P_lora × 2   // LoRA 可训练权重(FP16)
M_lora_optim  =  P_lora × 8   // AdamW: m(4B)+v(4B)=8B/param（仅 LoRA 参数需优化器）
M_lora_grad  =  P_lora × 2   // LoRA 梯度
P_lora  =  L × n_proj × 2 × r × H   // L=层数, n_proj=Target投影数, r=rank, 2=A+B

参数说明：

B_base基座权重精度FP16=2, INT8=1, INT4=0.5(QLoRA)
n_projTarget 投影数Q/K/V/O + Gate/Up/Down = 7（典型值）
rLoRA Rank通常 8~64

5.2   最低 GPU 数公式

▎ 终极公式：最低 GPU 数

N_min  =  ⌈ M_lora_total / (Mem_gpu × η_frag) ⌉

// LoRA 不需要 ZeRO，因为可训练参数极少，无需跨卡分片
// 但如果基座很大（>70B），需要 Tensor Parallel 分片基座权重

5.3   经验系数

η_frag = 0.85   同训练，显存碎片+框架开销。

M_overhead ≈ 2 GB   CUDA 上下文 + 框架缓冲。

M_act(act ckpt)   LoRA 激活值 = S × B × H × L × 2 × 2 bytes，与全量训练公式相同（开启 activation checkpointing，只存输入）。虽然 LoRA 可训练参数少，但前向传播仍需存中间激活用于反向传播。

5.4   QLoRA 省显存

策略基座精度M_frozen 占比典型节省
标准 LoRAFP16/BF16P_total × 2无节省
LoRA + INT8INT8 量化P_total × 1M_frozen 减半
QLoRA (INT4)INT4 + NF4P_total × 0.5M_frozen 省 75%

QLoRA 使用 NF4 量化 + 双量化，在几乎不损失精度的情况下将基座权重压缩至 4bit。这是单卡微调 70B 模型的关键技巧。

5.5   速算示例

场景：Hy3 混元3（295B 总参/21B 激活），8×H20，QLoRA INT4，r=16，n_proj=7，4K 序列

P_lora = 80 × 7 × 2 × 16 × 4096 ≈ 0.073B 参数（占总参数 0.025%）

M_frozen = 295B × 0.5 = 147.5 GB | M_lora = 0.073B × 2 = 0.15 GB | M_lora_optim = 0.073B × 8 = 0.59 GB

M_lora_grad = 0.073B × 2 = 0.15 GB | M_act ≈ 4096 × 1 × 4096 × 80 × 2 × 2 ≈ 5.0 GB | M_overhead ≈ 2 GB

M_total ≈ 147.5 + 0.15 + 0.59 + 0.15 + 5.0 + 2 = 155.4 GB

N_min = ⌈155.4 / (96 × 0.85)⌉ = ⌈1.90⌉ = 2 卡（单节点即可）

QLoRA 的威力：295B 模型 INT4 基座仅 147.5 GB，2 张 H20 即可微调。全精度 LoRA（FP16 基座）则需要 ⌈(590+0.9+5.0+2)/(96×0.85)⌉ = 8 卡。

Token 成本：推理成本速算公式 自建推理 · 成本拆解

给定 GPU 型号、集群规模和推理吞吐，速算自建推理的 Token 级成本。

6.1   核心推导逻辑

自建推理的成本拆解思路：把 GPU 租金（¥/小时）分摊到每个 token 上。

关键链路：GPU 时间成本 → 吞吐能力 → 每 token 分摊

核心洞察：tokens_per_sec 越高（推理越快），每 token 分摊到的 GPU 时间越少，成本越低。因此优化推理吞吐（量化、MoE、更大带宽）直接降低 Token 成本。

6.2   Token 成本公式

▎ 终极公式：Token 级推理成本

// Step 1: GPU 集群每秒花多少钱？
cost_per_sec  =  GPU_price × N_gpu / 3600   // GPU_price 单位: ¥/卡·小时

// Step 2: 集群每秒能输出多少 token？
tokens_per_sec  =  QPM × S_out / 60   // 从推理 QPM 公式获取

// Step 3: 每 token 分摊成本
cost_per_token  =  cost_per_sec / tokens_per_sec

// 展开：cost_per_token = (GPU_price × N_gpu) / (3600 × tokens_per_sec)

// Step 4: 常用换算
cost_per_1K  =  cost_per_token × 1000   // ¥/千 token（API 常用单位）
cost_per_M  =  cost_per_token × 1,000,000   // ¥/百万 token
cost_per_req  =  cost_per_token × S_out   // ¥/请求

// Step 5: 业务维度
daily_cost  =  cost_per_req × daily_requests
monthly_cost  =  daily_cost × 30

6.3   参数说明

GPU_price单卡租赁单价¥/卡·小时，各云厂商不同
N_gpuGPU 卡数单节点或多节点总卡数
tokens_per_sec集群总输出吞吐从推理 QPM 公式计算得出
QPM每分钟查询数由 Section 03 公式计算
S_out平均输出 token 数每请求的平均生成长度
daily_requests日请求量业务侧统计

6.5   成本优化杠杆

降低 cost_per_token 的核心杠杆（按影响力排序）：

杠杆原理效果
量化（INT4/INT8）减小每 token 权重读取量 → tokens/sec ↑2~4× 吞吐提升
MoE 架构P_active ≪ P_total → 每步读取大幅减少2~5× 吞吐提升
高带宽 GPU（H20）BW_mem 更大 → tokens/sec 线性增长1.5~2× vs 低带宽卡
Continuous Batching并行处理多请求，提升 GPU 利用率QPM 线性增长（受显存限制）
KV Cache 量化INT8 KV → 更多并发请求 → 更高批处理1.5~2× 并发空间
选择更便宜的 GPU降低 GPU_price 分子线性降本

6.6   速算示例

场景：8×H20，DeepSeek-V4-Flash（13B MoE 激活），INT8 量化，QPM≈102（来自 Section 03）

GPU_price = ¥24/h | N_gpu = 8 | S_out = 500 tokens

cost_per_sec = 24 × 8 / 3600 = ¥0.0533/s

tokens_per_sec = 102 × 500 / 60 = 850 tok/s

cost_per_token = 0.0533 / 850 = ¥0.0000627/token

cost_per_1K = ¥0.0627 | cost_per_M = ¥62.7 | cost_per_req = ¥0.0314

日 10 万请求: ¥3,135 | 月成本: ¥9.4 万

对比：同场景换 H100 SXM（¥52/h）

cost_per_sec = 52 × 8 / 3600 = ¥0.1156/s

tokens_per_sec ≈ 850 tok/s（H100 带宽低于 H20，实际吞吐可能更低）

cost_per_1K = ¥0.136 — H20 成本仅为 H100 的 46%

结论：对于 MoE 推理场景，H20 的 4 TB/s 带宽 + 低单价 = 最优性价比。H100 的算力优势在 decode 阶段带宽瓶颈下无法发挥。
