推理性能：QPM 速算公式 Decode 阶段 · 带宽瓶颈

给定模型参数与 GPU 型号，速算单节点能支撑的 QPM（每分钟查询数）。

3.1   核心推导逻辑

LLM 推理分两个阶段：

Prefill（预填充）：一次性处理输入 prompt，计算密集型，瓶颈在 FLOPs。

Decode（解码）：逐 token 自回归生成，每步都要重新读取全部权重+KV Cache，内存带宽密集型。

实际在线服务中，Decode 阶段占生成总耗时的 80-95%，因此 QPM 的瓶颈几乎总是内存带宽。

3.2   显存容量约束（能装下吗？）

单节点总显存  =  N_gpu × Mem_gpu

推理所需显存  =  M_weights + M_kv + M_overhead

M_weights  =  P_total × B_param   // MoE 模型：全部参数都要加载到显存
M_kv  =  2 × L × H_kv × D_head × S × B_concurrent × B_kv   // 2=K+V
M_overhead  ≈  2~4 GB   // CUDA 上下文 + 框架缓冲

推理（decode 阶段）的中间激活值是临时分配、前向完即释放的，框架内存池会复用，因此不占用稳态显存。激活值仅在 prefill 阶段处理长 prompt 时瞬态较大，但算完即释放，不影响"能否装下"的判断。

参数说明：

P_total模型总参数量（含 MoE 全部专家）如 1.6T（DeepSeek-V4-Pro）
B_param每参数字节数FP16=2, INT8=1, INT4=0.5
LTransformer 层数如 80
H_kvKV 头数（GQA 时较少）如 Llama3-70B 为 8
D_head每头维度通常 128
S上下文长度（token 数）可调变量，默认 4096
B_concurrent并发请求数取决于部署策略
B_kvKV Cache 精度字节数FP16=2, INT8=1

3.3   QPM 公式（Decode 阶段，带宽瓶颈）

▎ 终极公式：单节点 QPM（Decode 阶段，带宽瓶颈）

// 核心：Decode 每步读取 = 权重 + KV Cache
tokens/sec  =  N_gpu × BW_mem × η_hw × η_comm × η_moe  /  (P_active × B_param + KV_per_token × S_avg)

QPM  =  tokens/sec × 60  /  S_out

// 其中：
P_active  =  每 token 活跃参数量   // Dense: P_total；MoE: 激活专家参数量
KV_per_token  =  2 × L × H_kv × D_head × B_kv   // 每 token 的 KV Cache 字节数
S_avg  =  S_ctx + S_out/2   // 平均 KV 序列长度（prompt + 半程输出）
S_ctx  =  prompt 上下文长度   // 改上下文会改变 KV 读取量
S_out  =  平均每请求输出 token 数
B_param  =  每参数字节数   // FP16=2, INT8=1, INT4=0.5
η_hw  =  0.45~0.65   // 算子利用率（vLLM+FlashAttn ≈ 0.55）
η_comm  =  1 - (N_gpu-1)/(N_gpu×R_nvlink)   // 多卡通信效率
η_moe  =  Dense=1.0, MoE=0.60~0.85   // All-to-All 通信 + 专家负载不均衡 + 路由开销

经验系数说明

η_hw = 0.45~0.65   实测算子利用率。理论带宽不可能 100% 利用——FlashAttention 可达 60-70%，但 Continuous Batching、调度开销等因素拉低至 45-65%。vLLM 默认配置实测约 50-55%。取值建议：vLLM + FlashAttn 取 0.55，裸 PyTorch 取 0.40。

η_comm   多卡张量并行的通信效率。每步 AllReduce 需传 2×(N-1)/N 的激活数据，通信耗时与 NVLink/显存带宽比相关。8 卡 H100(NVLink 900GB/s vs 3.35TB/s) η_comm ≈ 0.87；8 卡 A800(NVLink 400GB/s vs 2TB/s) η_comm ≈ 0.78。

η_moe = 0.60~0.85   MoE 模型综合效率损失。包含三个因素：① All-to-All 通信：token 路由到持有激活专家的 GPU，结果再回收，延迟与 EP 度正相关；② 专家负载不均衡：192 专家选 top-8，不是每 GPU 均匀分到，有的空转有的过载；③ 路由门控开销：每层都要计算 gate + dispatch，虽小但不可忽略。取值建议：EP=8 取 0.70，EP=4 取 0.80，纯 TP(无 EP) 取 0.90。

3.4   MoE vs Dense 的区别

Dense 模型MoE 模型
P_total= P_active（全部参与计算）≥ P_active（专家权重全部加载，但只激活部分）
M_weightsP_active × B_paramP_total × B_param（显存装全量）
每 token 计算 FLOPs2 × P_active2 × P_active（只算激活参数）
每 token 读取带宽P_active × B_paramP_active × B_param（只读激活专家权重）
+ 极小的路由门控开销（可忽略）

MoE 关键：显存瓶颈看 P_total（全部专家需加载），计算/带宽瓶颈看 P_active（每 token 只读激活专家）。Decode 阶段每 token 权重读取量 = P_active × B_param，与 Dense 模型形式相同，只是 P_active ≪ P_total。若启用 Expert Parallelism，不同专家分到不同 GPU，等效每 GPU 读取量进一步降低。

3.5   速算示例

场景 A：8×H20，Qwen3-32B（32B Dense），INT8 量化，4K 上下文，输出 500 token

每步权重读取 = 32B × 1 byte = 32 GB | KV读取 ≈ 0.05 GB（可忽略）

tokens/sec = 8 × 4000 × 0.55 × 0.80 / 32 ≈ 440

QPM = 440 × 60 / 500 ≈ 53 QPM

场景 B：8×H20，DeepSeek-V4-Flash（284B/13B MoE），INT8 量化，4K 上下文，输出 500 token

每步权重读取 = 13B × 1 byte = 13 GB | KV读取 ≈ 0.02 GB（可忽略）

tokens/sec = 8 × 4000 × 0.55 × 0.80 × 0.70 / 13 ≈ 955

QPM = 955 × 60 / 500 ≈ 115 QPM

MoE 的 P_active(13B) 远小于 Dense 的 P_total(32B)，即使乘以 η_moe=0.70 的损耗，每步读取量仍只有 Dense 的 28%，所以 MoE 的 QPM 反而更高。这是 MoE 推理的核心优势：以 4.4× 参数量换取 2.2× 吞吐。CSA/HCA 注意力压缩后 KV 几乎可忽略，长上下文场景优势更大。

训练性能：最低 GPU 数量公式 全量训练 · AdamW + Mixed Precision

给定模型参数与 GPU 型号，速算全量训练需要的最低显卡数量。

4.1   全量训练显存拆解

▎ 全量训练显存  =  M_model + M_grad + M_optim + M_act

M_model  =  P_total × 2 bytes   // FP16/BF16 权重
M_grad  =  P_total × 2 bytes   // 梯度（同精度）
M_optim  =  P_total × 8 bytes   // AdamW: m(4B) + v(4B) = 8B/param（BF16 推荐，不需 master）
// ⚠️ 如用 FP16 训练(需 master weights): 请 ×1.5 = 12B/param M_act  =  S × B × H × L × 2 × 2 bytes   // 激活值(开启 activation checkpointing，只存输入)
// attention: 输入 S×B×H, FP16 = ×2 bytes | MLP: 输入 S×B×H, FP16 = ×2 bytes
// MoE 的 MLP 激活按 P_active/P_total 比例缩放

参数说明：

P_total模型总参数量MoE 按总参数计（含全部专家）
HHidden DimensionMoE 按激活参数的 hidden_dim
LTransformer 层数
S序列长度
B单 GPU Batch Size

4.2   最低 GPU 数公式

▎ 终极公式：最低 GPU 数

N_min  =  ⌈ M_train / (Mem_gpu × η_frag) ⌉

M_train  =  M_model + M_grad + M_optim + M_act   // ZeRO-0（无分片）
// ZeRO 分片后，M_grad 和 M_optim 按 N_gpu 均摊，详见 4.4

4.3   经验系数

η_frag = 0.85   显存碎片系数。实际可用显存 < GPU 标称值，CUDA 上下文 ~1.5GB、框架缓冲 ~1-2GB、碎片无法利用约 5-10%。综合取 0.85。

激活值(act ckpt)   公式中每层 ×2 表示：开启 activation checkpointing 后，每层只需存 attention 输入(2) + MLP 输入(2) = 2 项 × FP16(2 bytes) = 4 bytes/element。MoE 的 MLP 激活按 P_active/P_total 比例缩放。

4.4   ZeRO 策略显存节省

策略分片内容每 GPU 显存适用场景
ZeRO-0无分片M_total（完整副本）小模型 / 单卡
ZeRO-1优化器状态M_model + M_grad + M_optim/N中等模型
ZeRO-2优化器 + 梯度M_model + (M_grad+M_optim)/N大模型推荐
ZeRO-3全部分片(M_model+M_grad+M_optim)/N + M_act超大模型必需

ZeRO-3 的 M_act 不可分片（每 GPU 需完整激活），且额外通信开销约 15-20%。实际选择需平衡通信带宽与显存压力。

4.5   速算示例

场景：DeepSeek-V4-Flash（284B 总参/13B 激活），8×H20，ZeRO-3，4K 序列，BS=1

M_model = 284B × 2 = 568 GB | M_grad = 568 GB | M_optim = 284B × 8 = 2272 GB

M_act ≈ 4096 × 1 × 4096 × 43 × 2 × 2 ≈ 2.7 GB

M_total = 568 + 568 + 2272 + 2.7 = 3410.7 GB

ZeRO-3: 每 GPU ≈ (568+568+3408)/N + 49 + 568/N → N_min ≈ ⌈4593/(96×0.85)⌉ = ⌈56.3⌉ = 57 卡（建议 64 卡 = 8 节点）

MoE 训练：权重/梯度/优化器按 P_total=284B 计算，激活按 P_active 的 hidden_dim=4096 计算。

LoRA 微调：最低 GPU 数量公式 低秩适配 · 冻结基座

给定模型参数与 GPU 型号，速算 LoRA 微调需要的最低显卡数量。

5.1   LoRA 显存拆解

▎ LoRA 微调显存  =  M_frozen + M_lora + M_lora_optim + M_lora_grad + M_act + M_overhead

M_frozen  =  P_total × B_base   // 冻结基座权重（可量化）
M_lora  =  P_lora × 2   // LoRA 可训练权重(FP16)
M_lora_optim  =  P_lora × 8   // AdamW: m(4B)+v(4B)=8B/param（仅 LoRA 参数需优化器）
M_lora_grad  =  P_lora × 2   // LoRA 梯度
P_lora  =  L × n_proj × 2 × r × H   // L=层数, n_proj=Target投影数, r=rank, 2=A+B

参数说明：

B_base基座权重精度FP16=2, INT8=1, INT4=0.5(QLoRA)
n_projTarget 投影数Q/K/V/O + Gate/Up/Down = 7（典型值）
rLoRA Rank通常 8~64

5.2   最低 GPU 数公式

▎ 终极公式：最低 GPU 数

N_min  =  ⌈ M_lora_total / (Mem_gpu × η_frag) ⌉

// LoRA 不需要 ZeRO，因为可训练参数极少，无需跨卡分片
// 但如果基座很大（>70B），需要 Tensor Parallel 分片基座权重

5.3   经验系数

η_frag = 0.85   同训练，显存碎片+框架开销。

M_overhead ≈ 2 GB   CUDA 上下文 + 框架缓冲。

M_act(act ckpt)   LoRA 激活值 = S × B × H × L × 2 × 2 bytes，与全量训练公式相同（开启 activation checkpointing，只存输入）。虽然 LoRA 可训练参数少，但前向传播仍需存中间激活用于反向传播。

5.4   QLoRA 省显存

策略基座精度M_frozen 占比典型节省
标准 LoRAFP16/BF16P_total × 2无节省
LoRA + INT8INT8 量化P_total × 1M_frozen 减半
QLoRA (INT4)INT4 + NF4P_total × 0.5M_frozen 省 75%

QLoRA 使用 NF4 量化 + 双量化，在几乎不损失精度的情况下将基座权重压缩至 4bit。这是单卡微调 70B 模型的关键技巧。

5.5   速算示例

场景：Hy3 混元3（295B 总参/21B 激活），8×H20，QLoRA INT4，r=16，n_proj=7，4K 序列

P_lora = 80 × 7 × 2 × 16 × 4096 ≈ 0.58B 参数

M_frozen = 295B × 0.5 = 147.5 GB | M_lora = 0.58B × 2 = 1.16 GB | M_lora_optim = 0.58B × 8 = 4.64 GB

M_act ≈ 4096 × 1 × 4096 × 80 × 2 ≈ 2.7 GB | M_overhead ≈ 3 GB

M_total ≈ 147.5 + 1.16 + 4.64 + 1.16 + 2.7 + 3 = 160.2 GB

N_min = ⌈160.2 / (96 × 0.85)⌉ = ⌈1.96⌉ = 2 卡（单节点即可）

QLoRA 的威力：295B 模型 INT4 基座仅 147.5 GB，2 张 H20 即可微调。全精度则需要 ⌈(295×16+0.58×12+2.7+3)/(96×0.85)⌉ = 59 卡。
